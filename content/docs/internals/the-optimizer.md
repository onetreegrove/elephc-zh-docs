**源码目录：** `src/optimize/`

elephc 的优化器设计上保持简单，以 AST 为核心。它不构建独立的 IR，也不运行重量级的 SSA pass。相反，它执行一小组局部重写，这些重写已经能在生成的汇编质量和编译期可读性上带来明显收益。

目前优化器分为五个 pass：

1. `fold_constants(program)` — 在类型检查之前运行
2. `propagate_constants(program)` — 在类型检查成功后运行
3. `prune_constant_control_flow(program)` — 在传播和警告收集之后运行
4. `normalize_control_flow(program)` — 在裁剪之后运行，将结构等价的控制流壳层改写为更简单的 AST 形态
5. `eliminate_dead_code(program)` — 在规范化之后运行，从已规范化的 AST 中移除残留的不可达或不可观测语句

这种划分非常重要。某些重写仅凭语法即可安全执行，而另一些则只能在诊断已经检查过程序之后才能进行。

与这五个 pass 并行，优化器还会构建轻量级的本地**副作用摘要**（effect summaries）。这些摘要以保守方式回答两个问题：

- 该表达式是否具有可观测的副作用？
- 该表达式是否可能抛出异常？

正是这些副作用信息，使得后续的裁剪和死代码消除在处理 `try` / `catch`、可调用别名以及非平凡控制流合并时能够保持更高的精确性。

## 为什么在 AST 层面进行优化

elephc 直接从 AST 编译到目标汇编，中间没有用于优化的 IR，因此简化代码最经济高效的位置就是 AST 本身。

这带来了几个直接收益：

- 减少代码生成的工作量
- 生成的汇编更小、更清晰
- 对结果已知的表达式，减少运行时辅助函数调用
- 在不引入后端专用机制的前提下，提供一个保守的裁剪死分支的位置

示例：

```php
<?php
echo 2 ** 3;
echo "hello " . "world";
echo (int)"42";
```

到代码生成阶段时，这些表达式的结果已经是常量，无需调用 `pow`、`__rt_concat` 或数值字符串转换等运行时辅助函数。

## Pass 1：常量折叠

`fold_constants()` 递归遍历 AST，将仅凭子节点即可静态确定结果的表达式改写为常量。

当前支持的折叠范围包括：

- 标量算术：`+`、`-`、`*`、`/`、`%`、`**`
- 整数上的位运算和移位运算
- 一元 `-`、`!` 和 `~`
- 字符串字面量的 `.` 拼接
- 严格比较和数值比较
- 两侧均为标量常量时的逻辑 `&&` / `||`
- 飞船运算符 `<=>`
- 所选结果已知时的 `??`、三元表达式和 `match`
- 标量索引和关联数组字面量读取，如 `[2, 9][0]` 和 `["a" => 2]["a"]`（每个字面量项均为标量时）
- 语义明确时的标量类型转换，如 `(int)"42"` 或 `(bool)"0"`
- 在以下位置递归折叠：
  - 函数和方法体
  - 闭包和箭头函数
  - 默认参数值
  - 属性默认值
  - 常量声明

### 示例

```php
<?php
$x = (2 < 3) ? (2 ** 3) : (3 ** 4);
echo $x . "\n";
```

折叠后，AST 实际等价于：

```php
<?php
$x = 8;
echo $x . "\n";
```

## Pass 2：本地常量传播

`propagate_constants()` 在类型检查之后运行，此时类型检查器已对原始程序结构完成验证。

该 pass 仍刻意保持本地化和保守性。目前关注的场景包括：

- 直线代码中的标量局部变量赋值，例如：
  - `$x = 2;`
  - `$y = 3;`
  - `echo $x ** $y;`
- 每条穿透分支都同意相同标量值的简单 `if` 合并
- 所有可能出口路径都同意相同标量值的保守 `switch` 合并
- 已知主体的 `switch` 合并，仅模拟所选入口及其穿透后缀
- 每条可达穿透处理路径都同意相同标量值的保守 `try` / `catch` 合并
- 不抛出异常的 `try` 体，使 `catch` 中不可达的写入不进入 `try` 后的常量环境
- 识别来自局部合并表达式（如 `?:` 和 `match`）的统一标量赋值结果
- 识别通过 `list(...)` / `[...] = [...]` 解构固定标量数组字面量引入的标量局部变量
- 在简单循环中保留未改动的标量局部变量（保守的局部写分析能证明循环只修改其他变量时），包括简单嵌套的 `switch`、`try/catch/finally`、`foreach`、其他简单嵌套循环语句、局部数组写入如 `$items[] = $i` / `$items[0] = $i`、局部属性写入如 `$box->last = $i` / `$box->items[] = $i`、以及定向失效如 `unset($tmp)`，同时保留由 `for` 初始化子句引入的稳定标量值
- 已知 `while(false)`、`do...while(false)`、`while(true)` / `for(;;)` break 出口以及分支本地循环出口（当这些出口在标量值上达成一致时）的本地循环路径摘要
- 在完成替换后对表达式重新运行常量折叠
- 保守地传播到嵌套体，不尝试跨循环或通用路径敏感 CFG 求解完整数据流

### 示例

```php
<?php
if ($argc > 0) {
    $base = 2;
} else {
    $base = 2;
}

echo $base ** 3;
```

传播后，后续的 `echo` 实际等价于：

```php
<?php
echo 8.0;
```

这意味着后续 pass 无需再生成运行时 `pow` 路径。

## Pass 3：类型检查后的控制流裁剪

`prune_constant_control_flow()` 只在类型检查成功后运行。此时允许移除死分支和死语句，因为诊断已经看过了被检查的结构。

当前裁剪范围包括：

- 条件为常量的 `if` / `elseif` / `else` 链
- `while (false)`
- `do { ... } while (false)` 简化为体的单次执行
- `for (...; false; ...)`，保留 `init` 子句但移除死循环/更新工作
- 主体和模式静态可决定的 `match` 表达式
- 遮蔽的 `match` 臂和重复臂模式（当更早的臂已覆盖完全相同的模式条目时）
- 早期 case 前缀被证明不可能时的 `switch` 裁剪
- 以下语句后的不可达语句：
  - `return`
  - `throw`
  - `break`
  - `continue`
- 穷举 `if` / `else` 后的死代码
- 保守穷举 `switch ... default` 后的死代码
- 结果未被使用的纯表达式语句
- 以下位置内部的纯死子表达式：
  - 三元表达式
  - `??`
  - 短路 `&&` / `||`

## Pass 4：控制流规范化

`normalize_control_flow()` 在裁剪 pass 之后运行。此时 AST 已移除常量死分支，任务变为"将剩余控制流重塑为更简单但等价的形式"，而非"判断哪条分支是死的"。

当前规范化范围包括：

- 空的 `ifdef`、`if`、`switch` 以及退化的 `try` 壳层
- 单路径条件，例如：
  - `if ($cond) {} else { ... }` → `if (!$cond) { ... }`
  - 嵌套的单路径 `if` 链合并为一个带 `&&` 的条件
- `then` 和 `else` 体规范化为相同块的 `if` 语句，合并为"若条件可观测则求值，然后执行一次共享块"
- `elseif` 链规范化为嵌套的 `else { if (...) { ... } }` 形式
- 相邻且体相同的 `if` 链头合并为一个 `if ($a || $b) { ... }` 形式
- 相邻且回退相同的 `if` 链尾合并为一个 `if (!$a && $b) { ... } else { ... }` 形式
- 较长的 `if` 链反复规范化直至这些形式达到饱和
- 体相同的相邻 `switch` case 合并为单个多模式 case
- 纯穿透 `switch` 标签折叠到下一个非空 case 体
- 单个存活的 `switch` case 在可安全重建松散比较时改写为 `if`
- 体和变量相同的相邻 `catch` 子句合并为单个去重、稳定排序的多类型 catch
- 常量 `switch` 执行具体化为将要运行的精确语句尾，同时保留穿透和 `break`
- 不抛出异常的 `try` / `catch` 简化
- 外层 `finally` 块在恰好包裹一个自身没有 `finally` 的内层 `try` 时，折叠进该内层 `try`
- 将不抛出异常的穿透前缀安全提升出 `try` 块
- 当 `try` 体不能抛出且体穿透时，保守地展平 `try` / `finally`

### 示例

```php
<?php
try {
    echo "a";
    throw new Exception("boom");
} catch (Exception $e) {
    echo "b";
}
```

开头的 `echo "a";` 已知不会抛出异常，因此优化器可以将其提升到 `try` 块之外，只让真正可能抛出的尾部受处理器保护。

## Pass 5：死代码消除

`eliminate_dead_code()` 现在在规范化之后运行。此时 AST 已移除常量死分支并压缩了冗余控制流壳层，任务变为"丢弃剩余垃圾"，而非"重塑程序"。

当前死代码消除范围包括：

- 以下语句后的不可达语句：
  - `return`
  - `throw`
  - `break`
  - `continue`
- 穷举 `try/catch` 和 `try/finally` 出口后的语句
- DCE 后 `try` 体不再能抛出时，不可达的 `catch` 路径
- 遮蔽的 `catch` 子句（其异常类型已被更早的处理器完全覆盖，包括 `catch (Throwable ...)` 之后的所有后续处理器）
- 遮蔽的 `switch` 模式（其匹配点已被更早的 case 标签覆盖，包括完整 case 移除或穿透体合并（当没有入口模式剩余时））
- 当外层纯变量守卫或严格布尔检查已能确定嵌套分支结果时，对内部 `if` 区域进行裁剪，并在相关局部写入时失效守卫以保持保守性
- 基于守卫的裁剪现在也能理解简单的纯 `&&` / `||` 组合，因此无需先进行常量折叠即可移除 `if ($a && $b) { if (!$a || !$b) ... }` 之类的矛盾
- 松散相等和安全关系比较补集现在也纳入同一守卫模型，因此当外层分支证明矛盾时，可裁剪嵌套检查，如 `$x == 0` 后跟 `$x != 0`，或 `$x > 10` 后跟 `$x <= 10`
- 严格标量守卫现在纳入同一裁剪：在 `$x === null`、`$x === 0` 或 `$x === ""` 等检查之后，与已知精确值矛盾的嵌套区域可被移除
- 严格标量检查的负分支现在也贡献排除事实，因此 `$x === 0` 检查后的 `else` 路径或 `$x !== null` 真值路径，可在无需完整精确替换值的情况下裁剪嵌套矛盾
- 相同的严格标量守卫机制现在也覆盖精确浮点数以及 PHP falsy 字符串如 `""` 和 `"0"`，因此当这些值已知或已排除时，嵌套的真值检查和严格字面量矛盾可被裁剪
- 外层精确标量守卫现在也能裁剪不可能的 `switch` 入口：当 `switch ($x)` 的主体或 `switch (true|false)` 的守卫模式已被周围的严格检查确定时，死的头部 case 会被丢弃，CFG-lite pass 也可丢弃在选定精确入口之后不再有可达前驱的后续 `switch` 块
- `if` / `elseif` 链中的累积假守卫现在可以裁剪后续不可能的分支和不可达的 `else` 后缀，而无需将逻辑死尾拖过剩余流水线
- `switch (true|false)` 现在在 case 穿透中应用同样的累积守卫思路：当更早的无匹配路径已强制形成穷举结果时，后续守卫型 case 和 `default` 可被裁剪
- 直接入口的 `switch (true|false)` case 体也可从更早非穿透 case 继承累积无匹配守卫，从而裁剪后续 case 体内的嵌套矛盾，同时对穿透保持保守
- 多模式 `switch (true|false)` case 现在参与同一累积推理，因此一个 case 内的穷举标签集可移除后续死 case 和 `default`
- 精确标量守卫现在也驱动普通 `switch ($x)` 多模式 case 内的同一裁剪：不可能的标签会被丢弃，若某个存活的后续标签确保匹配，则后续死 case 和 `default` 也被移除
- 排除标量守卫现在也裁剪普通 `switch ($x)` 入口，因此即使 `$x` 的精确运行时值未知，外层事实如 `$x !== 1` 也可移除死的 `case 1:` 标签
- 真值事实现在也为普通 `switch ($x)` 裁剪 `case true` / `case false`：累积无匹配路径可消除死的布尔 case，甚至在剩余真值路径完全覆盖后移除死的 `default`
- 同一真值裁剪现在将更早的 `Unknown` 多模式入口保留为可达 CFG 入口点，从而避免过度裁剪前面的 case 体，同时仍移除死的布尔后缀和 `default`
- 真值事实也裁剪普通 `switch ($x)` 中相反真值的标量字面量标签，因此真值/假值外层守卫可在混合多模式 switch 内移除死的 `case 0`、`case ""`、`case null` 或类似真值字面量标签
- 使用单个守卫型模式的 `switch (true|false)` case 可在所选 case 体内进行同一内部区域裁剪，同样带有局部写入失效以保持保守性
- `catch` 和 `finally` 体现在仅对相关预处理路径上写入的局部变量使外层守卫事实失效，从而在该处保持健全的嵌套裁剪，同时不丢弃无关的守卫事实
- `switch` 的 throw 路径失效现在查询 CFG-lite 可达块集，因此不可能 case 体中的写入不会无必要地杀死 catch 体守卫，而在 `throw` 之前的可达 case 写入仍会使其失效
- catch 侧守卫失效现在是路径感知的：仅发生在非抛出 `try` 路径上的写入不再阻止 `catch` 内部的裁剪
- 仅包含可观测条件检查的空 `if` / `elseif` 链简化为最小形式
- 活跃链中间的空 `elseif` 体折叠为后续分支所需的最小否定守卫
- 尾部块后缀沉入 `if` 和 `ifdef` 穿透分支，使后续语句只在能到达时才保留
- 尾部块后缀沉入 `switch` 后缀（当后续代码确定可由最终可达路径穿透或从 case 经 `break` 退出时）
- 尾部块后缀沉入 `try` / `catch` 穿透路径，以及在保守情况下（每条 finally 前路径均必须穿透时）沉入 `finally`
- 不再指向可达工作的尾部空 `switch` 标签被丢弃
- 结果未被使用的纯表达式语句
- 被更早规范化暴露出的纯表达式语句

当前的路径感知 DCE 工作使用针对 `if`、`ifdef`、`switch` 和 `try` 的小型路径结果辅助函数，这些函数使用统一的本地尾路径词汇（`falls through`、`breaks`、`no tail`、`unknown`）。这使尾部沉入和壳层折叠能够共享同一个可达性模型，而无需为每种语句形态重复编写专用逻辑。

首批 `dead-code-elimination v3` 切片也开始将部分推理迁移到一个微型 CFG-lite 层。目前覆盖 `switch`、`if` 和 `try/catch/finally`：分支体被降低为小型基本块图，其尾部可达性从后继边分类，而非仅依赖手写扫描。这仍是 AST 局部的，不是完整的函数 CFG，但它是面向块感知 DCE 的第一步。

### 示例

```php
<?php
if (true) {
    echo "kept\n";
} else {
    echo pow(3, 4) . "\n";
}
```

裁剪和规范化后，死分支完全消失。最终的死代码 pass 需要检查的结构噪声更少，代码生成也不再生成 `pow` 路径。

## 副作用摘要：纯性与 `may_throw`

优化器现在维护一个小型本地副作用分析层，位于裁剪和死代码消除 pass 的下方。

当前覆盖范围包括：

- 已知纯/不抛出的内置函数，如 `strlen()`
- 体本身纯/不抛出的用户定义函数
- 使用相同保守摘要推断的用户定义静态方法
- 在 `$this` 上调用的私有实例方法（分派静态已知）
- 直接闭包调用和局部闭包别名
- 命名的一等可调用对象及对其的表达式调用
- 通过以下结构合并后存活的可调用别名：
  - `if` / `else`
  - `try` / `catch` / `finally`
  - `switch`
- 以下产生可调用对象的表达式：
  - 三元表达式
  - `??`
  - `match`
  （当每条存活分支都同意相同的可调用副作用时）

该分析仍刻意保持本地化。它不尝试求解通用的全程序纯性，而是专注于当前 AST 重写中最重要的一小组调用形态。

### 示例

```php
<?php
$f = match ($mode) {
    1 => strlen(...),
    default => strlen(...),
};

try {
    echo $f("abc");
} catch (Exception $e) {
    echo pow(2, 8);
}
```

由于每个 `match` 臂都产生相同的已知纯/不抛出可调用对象，优化器可以证明 `catch` 路径是死代码，从而完全避免生成 `pow` 分支。

## 为什么有五个 pass

如果 elephc 在类型检查之前就移除整个分支，可能会意外隐藏有用的诊断信息。

例如：

```php
<?php
if (false) {
    $x = "hello";
    $x = 123;
}
```

从优化角度看，该块是死代码。但从编译器用户体验角度看，类型检查器和警告 pass 在任何激进裁剪发生之前看到它，仍然是有价值的。

因此当前规则是：

- 尽早折叠明显的纯标量表达式
- 仅在类型检查后传播已知标量局部变量
- 仅在类型检查后裁剪较大的死控制流
- 将剩余控制流规范化为更简单的等价形式
- 仅在上述更早的 pass 已简化语法树后，运行结构性死代码清理

## 保守性与副作用

优化器对"纯"或"不抛出"的判定刻意保持保守。

它现在能精确识别有用的调用表达式子集，但对于以下广泛的动态操作，**不**假设其纯性：

- 未知函数或方法调用
- 超出静态已知 `$this` / 私有方法范围的动态实例分派
- 对象创建
- 运行时钩子或动态行为可能介入的大多数属性和数组读取
- buffer 分配
- 自增/自减
- `throw`

这种保守性是该 pass 默认安全运行的原因：如果一个表达式可能存在运行时行为，而 elephc 无法通过本地摘要证明其不存在，优化器会选择保留它。

## 管道运算符优化

PHP 8.5 的管道运算符 `|>` 被实现为专用的 `ExprKind::Pipe` 节点，而非 `BinOp`，这使优化器能将其作为一等调用点来推理，而非不透明的二元操作。端到端共应用三层优化：

### 副作用建模

`src/optimize/effects.rs` 中的 `ExprKind::Pipe { value, callable }` 组合了 `value` 的副作用、`callable` 的副作用以及通过 `expr_call_effect` **调用** `callable` 的副作用。由于当可调用对象是引用纯内置函数（`strlen`、`strtoupper`……）的一等可调用引用时，每目标调用副作用已折叠为 `Effect::PURE`，纯管道表达式语句在可观测上是死代码，DCE pass 会将其移除。不会额外应用 `with_side_effects()` 包装——那样会遮蔽目标的精确副作用并阻碍 DCE。

### 纯管道的常量折叠

`src/optimize/fold/pipes.rs::try_fold_pure_pipe` 从 `fold_expr` 的 Pipe 分支调用。当值为标量字面量（或用于谓词的数组字面量），且可调用对象为引用白名单纯内置函数的 `FirstClassCallable(Function(name))` 时，管道在编译期折叠为字面量。示例：

- `"hello" |> strlen(...)` → `5`
- `3.7 |> floor(...)` → `3.0`
- `"hello" |> strtoupper(...) |> strrev(...)` → `"OLLEH"`（每个阶段折叠后传入下一阶段）
- `5 |> is_int(...)` → `true`
- `5 |> gettype(...)` → `"integer"`

白名单刻意保持狭窄：仅包含 Rust 实现与 PHP 在我们接受的字面量类型上字节等价的内置函数。非 ASCII 字符串、`NaN`/无穷大浮点数以及 `abs` 的 `i64::MIN` 回退到运行时调用，以保留 PHP 语义。

### 一等可调用短路（代码生成侧）

从优化器角度追踪，因为类型检查器的 `first_class_callable_targets` 映射被镜像到代码生成 `Context` 中，用于在调用点绕过闭包包装器。`$cb = foo(...)` 后跟 `$x |> $cb` 会降低为直接的 `bl _fn_foo`，而非通过 FCC 包装器的间接调用。同样的短路对管道运算符之外的 `$cb(args)` 也生效——`array_map`、`call_user_func` 或普通的 `$cb()` 调用均受益。

`self::method(...)` 和 `parent::method(...)` 在 FCC 变量存储时预解析为 `Named(class)`，因此短路也适用于它们。`static::method(...)` 在存储中保留其 `Static` 接收者，调用点复用调用者作用域的 `__elephc_fcc_called_class_id` / `__elephc_called_class_id` / `__elephc_fcc_this` / `$this` 链——与闭包包装器查询的链完全相同，只是省去了包装器蹦床。

### 死包装器桩化

当一等可调用对象被赋值给某个局部变量，且该变量的每次读取都被短路（变量从未到达 `emit_variable`、`emit_closure_call` 的回退路径、`call_user_func`、`array_map`、`Fiber::new` 等）时，延迟的包装器体被替换为一个微型 `mov #0; ret` 桩。赋值点的地址加载保持可解析，槽仍接收一个值，而二进制文件则去掉了每个未调用包装器约 50–100 条死 prologue/body/epilogue 指令。`Context::mark_fcc_used` 是每条"此 FCC 值逃逸"路径调用的中心钩子。

## 优化器尚未实现的功能

当前优化器仍刻意保持本地化。尚未实现：

- 跨任意循环和通用路径合并的完整不动点/基本块常量传播
- 跨堆支持局部变量和更广泛别名情况的更丰富内存模型感知传播
- 精确异常类型可达性、嵌套重抛出建模，以及超出当前路径感知 `try` 启发式的更少保守 `finally` 失效
- 超出当前布尔、标量、松散比较和安全关系补集事实的更广泛守卫推理（范围事实和多变量关系）
- 超出当前本地 AST 壳层重写的更广泛控制流规范化
- 后端专用窥孔清理
- 当包装器被桩化时，FCC 赋值点处 `adrp/add/stur` 指令三元组的消除（桩地址仍被加载和存储，尽管两者均是死代码）

这些仍是后续优化工作的路线图项目。

注意，**寄存器分配**不再在此列表中：EIR 后端现在运行一个线性扫描寄存器分配器（`src/ir_passes/`），详见 [The IR](the-ir.md)。它是 EIR 上的后端 pass，而非 AST 级优化，因此位于 `src/optimize/` 之外。

IR 级变换也不再在此处。EIR 后端在降低后运行一个不动点 pass 驱动器（`src/ir_passes/driver.rs`），从恒等算术折叠（`x + 0`、`x * 1`、`x ^ x`……）和本地窥孔模式（box/unbox 消除、标量 load/store 转发、成对 acquire/release 消除、字符串字面量拼接折叠、冗余 `move`/`borrow` 清理）开始，接着是逐块常量折叠（将操作数均为编译期常量的运算折叠为单个常量，如 `5 * 5` → `25`、`0 < 5` → `true`）——与窥孔的标量 load/store 转发组合后，可通过 EIR 值 ID 和局部槽传播常量——然后是支配感知的公共子表达式消除（复用每条路径上均可用的纯计算，按块内及跨块通过支配树值编号）、循环不变代码外提（将纯循环不变计算提升到循环前置块）、CFG 感知的未使用纯结果产生 EIR 指令死指令消除、CFG 感知的死存储消除（针对在被覆写前从未被读取的标量 PHP 局部槽的 `store_local` 写入），以及分支简化（常量条件 `cond_br`/`switch` 折叠、空块跳转线程化和不可达块中性化）。这些是 AST 优化器难以表达的重写，因为它们需要值标识、基本块、活跃性或支配关系；参见 [Optimization Passes](the-ir.md#optimization-passes)。
